# 项目面试问题（Day 76）

这份问题清单用于准备 Knowledge Agent 的项目深挖。回答时不要背概念，尽量按“项目场景 -> 具体实现 -> 为什么这样设计 -> 当前边界”来讲。

## 项目总览

### 1. 你这个项目是做什么的？

这是一个企业知识库 RAG + Agent 系统。用户可以上传 PDF、txt、markdown 文档，系统会解析文本、切分 chunks、生成 embedding，并保存到 PostgreSQL + pgvector。用户提问时，系统会先检索相关文档片段，再调用 LLM 生成带引用来源的回答。后面我还加了工具调用、权限控制、trace 追踪和离线评测，让它不只是 demo，而是更接近真实工程项目。

### 2. 这个项目和普通 ChatGPT 问答有什么区别？

普通 ChatGPT 主要依赖模型已有知识，回答不一定来自用户上传的资料。这个项目的回答必须基于已上传文档，先检索相关 chunks，再把这些 chunks 作为上下文交给模型。回答会返回文档名、页码、chunk_id 和原文片段，用户可以检查答案来源。这样项目重点不是“模型会不会说”，而是“答案能不能追溯”。

### 3. 这个项目的整体架构怎么讲？

前端是 React 工作台，负责上传文档、提问、展示引用和 trace。后端是 FastAPI，按接口分成文档、问答、trace 等入口。业务层负责文档解析、chunk 切分、向量检索、RAG 生成、Agent 工具调用和运行追踪。存储层用 PostgreSQL 保存结构化数据，用 pgvector 保存 chunk embedding。

### 4. 为什么选择 PostgreSQL + pgvector？

这个项目既需要保存文档元数据、会话、消息、日志，也需要做向量检索。PostgreSQL 可以处理关系型数据，pgvector 又能在同一个数据库里保存 embedding 并做相似度搜索。这样可以少维护一个独立向量库，项目部署和排查都更简单。对于 3 个月冲刺项目来说，这个组合足够真实，也不会把复杂度拉得太高。

## RAG 深挖

### 5. 文档上传后，系统内部发生了什么？

用户上传文件后，请求先进入 `POST /api/documents/upload`。系统会校验文件类型、大小和空内容，然后把原文件保存到 `storage/uploads`。之后解析文本，把页面文本切分成 chunks，并为每个 chunk 生成 embedding。最后把文档元数据、chunk 原文、页码、chunk_index 和 embedding 写入数据库。

### 6. chunk size 和 overlap 是什么？你为什么需要它们？

chunk size 是每个文本片段的大致长度，overlap 是相邻片段之间保留的重复内容。如果 chunk 太大，检索结果会带入很多无关信息；如果太小，答案需要的上下文可能被切断。overlap 可以减少关键信息刚好落在切分边界时丢失的问题。这个项目后面用评测结果对 chunk size、top_k 和阈值做对比，而不是凭感觉调参。

### 7. 用户提问后，RAG 链路怎么走？

用户问题进入 `POST /api/chat` 后，系统先把问题转成 embedding。然后用这个问题向量去 pgvector 检索最相似的 chunks。检索结果会被整理成上下文，和用户问题、回答规则一起拼成 prompt 交给 LLM。模型回答后，系统再把 sources、耗时、run_id 一起返回给前端。

### 8. 你怎么做引用来源？

检索时每个 chunk 都带有文档名、页码、chunk_id、片段内容和相似度距离。生成回答后，后端不会只返回 answer，还会返回 `sources` 列表。前端展示回答时，同时展示这些 sources，用户可以看到答案依据来自哪份文档、哪一页。这样面试时可以强调：引用不是模型自己编的，而是检索结果里保留下来的结构化信息。

### 9. 如果检索不到资料怎么办？

项目里有拒答机制，不会把检索不到的问题直接交给模型自由发挥。当相似度距离超过阈值，或者没有可用 chunks 时，系统会返回“我在已上传文档里没有找到足够信息。”。另外输出检查会检查确定性回答是否有 sources，如果没有引用来源，也会拒答或拦截。这样可以降低模型幻觉和无来源回答的风险。

### 10. RAG 里 prompt 是怎么设计的？

prompt 会包含用户问题、检索到的文档片段，以及回答规则。规则会要求模型基于资料回答，不能脱离上下文编造，并且无法回答时要明确拒答。这样做的目标不是让 prompt 很花哨，而是让模型知道哪些内容是证据、哪些行为是不允许的。后续如果发现回答缺引用或越界，就可以继续强化 prompt 和输出检查。

## Agent 深挖

### 11. 你的 Agent 和 RAG 问答有什么区别？

RAG 问答是一条固定链路：问题 -> 检索 -> 生成。Agent 则多了一层工具调用能力，它可以根据问题选择不同工具，比如列出文档、检索文档、总结文档或创建笔记。这个项目里的 Agent 不是无限自主执行，而是一个有边界的工具调用主循环。它的重点是展示工具设计、参数校验、权限控制和 trace 记录。

### 12. 你实现了哪些工具？

当前工具包括 `list_documents()`、`retrieve_documents(query)`、`summarize_document(document_id)` 和 `create_note(title, content, source_ids)`。前三个是只读工具，只查询或总结资料，不修改数据库。`create_note` 是写入工具，会新增笔记，所以需要用户确认。删除文档这类危险工具在 Agent 里不开放。

### 13. 为什么工具要分权限等级？

Agent 能调用工具后，就不能默认所有工具都安全。只读工具通常可以直接执行，但写入工具会改变数据库，危险工具可能删除数据或访问敏感资源。这个项目用 Tool Registry 把工具分成只读、写入、危险三级，并结合会话白名单控制可用工具。这样即使模型想调用未开放工具，也会被权限检查拦截。

### 14. create_note 为什么需要用户确认？

`create_note` 会写入数据库，属于会改变系统状态的操作。如果模型误判用户意图，直接创建笔记就可能产生垃圾数据或错误记录。项目里把它定义为写入工具，执行前必须带 confirmed 标记。这个设计体现的是 Agent 工程里的安全边界：查询可以自动化，写入要谨慎。

### 15. 工具参数错误时怎么处理？

工具参数使用 Pydantic schema 做校验。比如需要 document_id 的工具，如果传入了错误类型或缺少必要字段，就不会真正执行工具。系统会返回参数校验失败，并把失败记录到 trace 或日志里。这样问题可以定位到“工具参数错”，而不是误以为是模型或数据库坏了。

## 工程排查

### 16. trace 在项目里解决什么问题？

trace 解决的是“只看最终回答不知道哪里出错”的问题。一次问答会生成一个 run_id，模型调用、工具调用、输出检查等步骤都会记录输入摘要、输出摘要、耗时和状态。前端可以按 run_id 查询 `agent_traces`，看到这次请求每一步发生了什么。排查时就能判断问题在检索、工具、模型、引用检查还是性能耗时。

### 17. 如果用户说回答不对，你怎么排查？

我会先拿这次回答的 run_id 查 trace，确认检索是否命中了正确 chunks。如果检索没命中，就看问题向量、chunk 切分、top_k 和阈值；如果检索命中了但回答不对，就看 prompt 和模型输出。还要检查 sources 是否为空，以及输出检查有没有拦截。这样排查顺序是从数据和链路开始，而不是直接猜模型错了。

### 18. 这个项目如何处理 prompt injection？

项目里有输入 Guardrails，会拦截明显的越权或提示词注入风险。比如用户要求忽略系统规则、绕过权限、执行删除类操作，就不应该继续进入工具调用。Agent 工具层还有权限等级和白名单，即使输入检查漏掉一部分，工具执行前也会再次校验。它不是绝对安全系统，但展示了多层防护思路。

## 评测和优化

### 19. 你怎么证明 RAG 效果变好了？

我没有只靠手动问几个问题判断效果，而是建立了固定评测集。评测脚本会批量调用 RAG 接口，并保存每条问题的结果。分析脚本会统计 top-3 检索命中率、引用完整率和失败原因。优化 chunk size、top_k 或阈值时，用同一批问题前后对比，才能说明效果是否真的改善。

### 20. 这个项目还有哪些可以继续改进的地方？

第一，可以加入 rerank，把向量检索结果再排序，提高复杂问题的命中质量。第二，可以做更完整的 Docker Compose 一键部署，把后端、前端和数据库编排起来。第三，可以扩展评测集数量，并增加回答正确性评估。第四，可以把 Agent 的工具选择从规则判断升级为更标准的 tool calling 流程。
